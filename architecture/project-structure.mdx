## SvelteKit file-based routing

SvelteKit uses a convention-over-configuration approach to routing. The filename determines both the URL and the role of each file.

<AccordionGroup>
  <Accordion title="+page.svelte ‚Äî route components">
    Each `+page.svelte` file maps to a URL. A file at `src/routes/contact/+page.svelte` renders at `/contact/`. The component receives data from an optional co-located `+page.js` or `+page.server.js` loader.
  </Accordion>
  <Accordion title="+layout.svelte ‚Äî shared shell">
    A `+layout.svelte` wraps every route in the same directory and all nested routes. The root `src/routes/+layout.svelte` therefore wraps every page on the site. It renders the `<Nav>` component, handles locale loading, and manages the View Transition API.
  </Accordion>
  <Accordion title="+layout.js ‚Äî layout configuration">
    The root `src/routes/+layout.js` sets two SvelteKit options that apply to the entire site:

    ```javascript
    import '$lib/locales/i18n.js';

    export const prerender = true;
    export const trailingSlash = 'always';
    ```

    `prerender = true` tells SvelteKit to generate all pages as static HTML at build time. `trailingSlash = 'always'` ensures consistent URL canonicalisation on GitHub Pages and Cloudflare Pages.
  </Accordion>
  <Accordion title="+error.svelte ‚Äî custom error page">
    Renders whenever SvelteKit encounters an unhandled error or a 404. Placed at the root of `src/routes/` so it applies site-wide.
  </Accordion>
</AccordionGroup>

## `src/lib/state.svelte.js` ‚Äî reactive state

Shared reactive state that needs to be accessible across components without prop-drilling is declared here using Svelte 5's `$state` rune.

```javascript
export const themeState = $state({
    accentColor: null,
    accentRgb: null
});
```

The `$state` rune creates a reactive object at module scope. Any component that imports `themeState` and reads from it will automatically re-render when the values change. `accentColor` holds the current hex value of the user's chosen accent color, and `accentRgb` holds its pre-computed `r, g, b` string for use with Tailwind's `rgb()` utilities and CSS `opacity` modifiers.

<Note>
  Because this file uses the `.svelte.js` extension, Svelte's compiler processes it and the `$state` rune is valid outside of a `.svelte` component file.
</Note>

## `src/hooks.server.js` ‚Äî server entry point

SvelteKit calls the exported `handle` function on every request before the framework renders a page. This file has two responsibilities:

1. **FOUC prevention** ‚Äî reads the `accent_color` cookie and injects the correct CSS variables into the HTML shell *before* it reaches the browser, so the accent color is never wrong on first paint.
2. **Security headers** ‚Äî sets a strict set of response headers on every route.

```javascript
export const handle = async ({ event, resolve }) => {
    const accentColor = event.cookies.get('accent_color') || '#e16666';

    const hexToRgbStr = (hex) => {
        if (!hex || hex.length < 7) return '225, 102, 102';
        const r = parseInt(hex.slice(1, 3), 16);
        const g = parseInt(hex.slice(3, 5), 16);
        const b = parseInt(hex.slice(5, 7), 16);
        return `${r}, ${g}, ${b}`;
    };

    const accentRgb = hexToRgbStr(accentColor);

    const response = await resolve(event, {
        transformPageChunk: ({ html }) =>
            html
                .replace('%accent%', accentColor)
                .replace('%accent_rgb%', accentRgb)
    });

    response.headers.set('X-Frame-Options', 'SAMEORIGIN');
    // ... additional security headers

    return response;
};
```
