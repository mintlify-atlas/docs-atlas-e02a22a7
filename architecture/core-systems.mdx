## Accent color system

The accent color is a user-configurable hex value stored in a browser cookie (`accent_color`). Because the site is statically generated, the challenge is making sure the correct color appears on the *very first paint* — before any JavaScript executes — so there is no flash of the default color.

The solution has three layers that execute in sequence:

<Steps>
  <Step title="Server: cookie → CSS variable injection">
    `src/hooks.server.js` runs on every request. It reads the `accent_color` cookie, converts the hex value to an RGB triple, then uses `transformPageChunk` to replace the `%accent%` and `%accent_rgb%` placeholders in `app.html` before the HTML reaches the browser.

    ```javascript
    export const handle = async ({ event, resolve }) => {
        const accentColor = event.cookies.get('accent_color') || '#e16666';

        const hexToRgbStr = (hex) => {
            if (!hex || hex.length < 7) return '225, 102, 102';
            const r = parseInt(hex.slice(1, 3), 16);
            const g = parseInt(hex.slice(3, 5), 16);
            const b = parseInt(hex.slice(5, 7), 16);
            return `${r}, ${g}, ${b}`;
        };

        const accentRgb = hexToRgbStr(accentColor);

        const response = await resolve(event, {
            transformPageChunk: ({ html }) =>
                html
                    .replace('%accent%', accentColor)
                    .replace('%accent_rgb%', accentRgb)
        });

        return response;
    };
    ```

    If no cookie exists, the fallback color `#e16666` (a muted red) is used.
  </Step>

  <Step title="HTML shell: CSS variable declaration">
    `src/app.html` declares the two CSS variables in a `<style>` block inside `<head>`. The placeholders are the injection points targeted by `transformPageChunk`.

    ```html
    <style>
      :root {
        --accent: %accent%;
        --accent-rgb: %accent_rgb%;
      }
    </style>
    ```

    Because this block is part of the static HTML, the variables are defined before any stylesheet or script parses, eliminating FOUC entirely.
  </Step>

  <Step title="Client: inline script guard for new visitors">
    `app.html` also includes an inline IIFE that runs immediately on parse. If the cookie already exists, it re-applies the values to `document.documentElement` to guarantee consistency during client-side hydration. If no cookie exists (a new visitor), `+layout.svelte` generates a random light-range color on `onMount` and writes the cookie for future visits.

    ```javascript
    // app.html — inline IIFE
    (function() {
        const getCookie = (name) => {
            const value = `; ${document.cookie}`;
            const parts = value.split(`; ${name}=`);
            if (parts.length === 2) return parts.pop().split(";").shift();
            return null;
        };
        const hexToRgbStr = (hex) => {
            const cleanHex = decodeURIComponent(hex).replace('#', '');
            if (cleanHex.length < 6) return '225, 102, 102';
            const r = parseInt(cleanHex.slice(0, 2), 16);
            const g = parseInt(cleanHex.slice(2, 4), 16);
            const b = parseInt(cleanHex.slice(4, 6), 16);
            return `${r}, ${g}, ${b}`;
        };
        const savedColor = getCookie('accent_color');
        if (savedColor) {
            const color = decodeURIComponent(savedColor);
            document.documentElement.style.setProperty('--accent', color);
            document.documentElement.style.setProperty('--accent-rgb', hexToRgbStr(color));
        }
    })();
    ```
  </Step>

  <Step title="New visitor: random color assignment">
    When no `accent_color` cookie is present, `+layout.svelte`'s `onMount` generates a random color biased toward light values (each channel between 128–255) and persists it for one year.

    ```javascript
    // src/routes/+layout.svelte — onMount
    if (!getCookie("accent_color")) {
      const pickLight = () =>
        Math.floor(128 + Math.random() * 127)
          .toString(16)
          .padStart(2, "0");
      const randomColor = `#${pickLight()}${pickLight()}${pickLight()}`;
      document.cookie = `accent_color=${encodeURIComponent(randomColor)}; max-age=31536000; path=/; samesite=lax`;

      document.documentElement.style.setProperty("--accent", randomColor);
      document.documentElement.style.setProperty(
        "--accent-rgb",
        hexToRgbStr(randomColor),
      );
    }
    ```
  </Step>
</Steps>

### Security headers

After `transformPageChunk` resolves, `hooks.server.js` attaches a hardened set of HTTP response headers to every page response:

```javascript
response.headers.set('X-Frame-Options', 'SAMEORIGIN');
response.headers.set('X-Content-Type-Options', 'nosniff');
response.headers.set('Referrer-Policy', 'strict-origin-when-cross-origin');
response.headers.set('Permissions-Policy', 'camera=(), microphone=(), geolocation=()');
response.headers.set(
    'Content-Security-Policy',
    [
        "default-src 'self'",
        "script-src 'self' 'unsafe-inline'",
        "style-src 'self' 'unsafe-inline' https://fonts.googleapis.com",
        "font-src 'self' https://fonts.gstatic.com",
        "img-src 'self' data: https://github.com https://*.githubusercontent.com https://cdn.stats.fm https://*.stats.fm https://i.scdn.co https://mosaic.scdn.com https://is1-ssl.mzstatic.com https://www.transparenttextures.com",
        "connect-src 'self' https://beta-api.stats.fm https://api.github.com",
        "frame-ancestors 'none'"
    ].join('; ')
);
```

| Header | Value | Purpose |
|---|---|---|
| `X-Frame-Options` | `SAMEORIGIN` | Prevents clickjacking by disallowing cross-origin iframes. |
| `X-Content-Type-Options` | `nosniff` | Stops browsers from MIME-sniffing responses away from the declared type. |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Sends the full URL as referrer for same-origin requests; only the origin for cross-origin. |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | Explicitly revokes browser feature access the site does not need. |
| `Content-Security-Policy` | See above | Whitelists allowed sources for scripts, styles, fonts, images, and API connections. |

<Warning>
  The CSP includes `unsafe-inline` for scripts and styles, which is required for SvelteKit's hydration and Tailwind's inline styles. Avoid adding additional inline scripts outside the framework.
</Warning>

## Internationalization

Localization is managed by `svelte-i18n`. The library exposes a reactive `locale` store and a `$t` translation function used throughout all page components.

### Locale registration

`src/lib/locales/i18n.js` registers all four supported locales at module initialization time:

```javascript
import { addMessages, init, getLocaleFromNavigator, locale } from 'svelte-i18n';
import enMessages from './data/en.json';
import itMessages from './data/it.json';
import deMessages from './data/de.json';
import esMessages from './data/es.json';

addMessages('en', enMessages);
addMessages('it', itMessages);
addMessages('de', deMessages);
addMessages('es', esMessages);
```

Supported locales: `en` (English), `it` (Italian), `de` (German), `es` (Spanish).

### Locale detection order

On the client, the desired locale is resolved by checking sources in priority order:

<Steps>
  <Step title="Cookie — highest priority">
    The `locale` cookie is read first. If it contains a supported locale code, that locale is used. The cookie persists for one year (`max-age=31536000`) so returning visitors always see the language they last selected.

    ```javascript
    const getCookie = (name) => {
      if (typeof document === 'undefined') return null;
      const value = `; ${document.cookie}`;
      const parts = value.split(`; ${name}=`);
      if (parts.length === 2) return parts.pop().split(';').shift();
      return null;
    };
    ```
  </Step>
  <Step title="Browser navigator — fallback">
    If no cookie exists, `getLocaleFromNavigator()` reads the browser's `navigator.language` value (e.g. `it-IT`). The `normalizeLocale` helper strips the region tag and checks the result against `supportedLocales`.

    ```javascript
    const supportedLocales = new Set(['en', 'it', 'de', 'es']);

    const normalizeLocale = (value) => {
      if (!value || typeof value !== 'string') return 'en';
      const normalized = value.toLowerCase().split('-')[0];
      return supportedLocales.has(normalized) ? normalized : 'en';
    };
    ```
  </Step>
  <Step title="Hard fallback — 'en'">
    If the navigator locale is unsupported or unavailable, the system falls back to English. This is also the `fallbackLocale` passed to `init()`, so any missing translation key in a non-English locale renders its English equivalent.

    ```javascript
    const desiredClientLocale =
      typeof window !== 'undefined'
        ? normalizeLocale(getCookie('locale') || getLocaleFromNavigator())
        : 'en';

    const initialLocale = 'en';

    init({
      fallbackLocale: 'en',
      initialLocale,
    });
    ```
  </Step>
</Steps>

### Hydration-safe locale application

The server always renders with `initialLocale: 'en'` to produce stable HTML. The client applies the desired locale after hydration inside a `requestAnimationFrame` callback, which avoids Svelte's `hydration_html_changed` warning:

```javascript
if (typeof window !== 'undefined') {
  locale.subscribe((newLocale) => {
    if (newLocale) {
      document.cookie = `locale=${newLocale}; max-age=31536000; path=/; samesite=lax`;
    }
  });

  // Apply saved locale after hydration to avoid hydration_html_changed warnings.
  if (desiredClientLocale !== initialLocale) {
    requestAnimationFrame(() => {
      locale.set(desiredClientLocale);
    });
  }
}
```

The `locale.subscribe` block simultaneously persists any new locale selection back to the cookie whenever the store value changes.

### Translation file structure

Each locale has a corresponding JSON file under `src/lib/locales/data/`. All keys are namespaced by route:

```json
{
  "nav": {
    "home": "home",
    "projects": "projects",
    "uses": "/uses",
    "contact": "contact"
  },
  "home": {
    "greeting": "hi, i'm {name}.",
    "who_am_i": "who am i?"
  },
  "projects": {
    "header": "the archive.",
    "description": "a collection of things i've built..."
  },
  "contact": { ... },
  "uses": { ... }
}
```

<Note>
  The `uses.categories` key holds an array of objects (each with `title` and `items`). This is the only key in the schema that is not a flat string — it drives the entire structured list on the `/uses` page.
</Note>

## View Transition API

Page navigations on desktop use the browser's native [View Transition API](https://developer.mozilla.org/en-US/docs/Web/API/View_Transitions_API) for smooth animated transitions between routes.

### Implementation

The transition is wired in `src/routes/+layout.svelte` using SvelteKit's `onNavigate` hook:

```javascript
onNavigate((navigation) => {
    if (!document.startViewTransition) return;

    const isMobile = window.innerWidth < 768;
    if (isMobile) return;

    return new Promise((resolve) => {
      document.startViewTransition(async () => {
        resolve();
        await navigation.complete;
      });
    });
});
```

### How it works

<Steps>
  <Step title="Feature detection">
    `if (!document.startViewTransition) return;` exits immediately on browsers that do not support the API (Firefox without the flag, older Safari). Navigation continues normally with no animation.
  </Step>
  <Step title="Mobile guard">
    `if (window.innerWidth < 768) return;` disables transitions on viewports narrower than 768 px. On mobile, the transition overhead is not worth the cost — instant navigation feels faster.
  </Step>
  <Step title="Promise integration with SvelteKit">
    `onNavigate` accepts a callback that may return a Promise. By returning a Promise and resolving it inside `startViewTransition`, the code tells SvelteKit to pause the DOM swap until the browser has captured the "before" screenshot, then complete the navigation (`await navigation.complete`) so the browser can capture the "after" screenshot and play the crossfade.
  </Step>
</Steps>

### Scroll reset

After every navigation completes, the scroll position is reset to the top of the page instantly:

```javascript
afterNavigate(() => {
    window.scrollTo({ top: 0, behavior: "instant" });
});
```

Using `behavior: "instant"` rather than `"smooth"` avoids a jarring double-animation when a view transition and a smooth scroll would otherwise run at the same time.

<Tip>
  The View Transition API's default animation is a simple crossfade. Custom transition styles can be added in `app.css` using the `::view-transition-old` and `::view-transition-new` pseudo-elements.
</Tip>
