---
title: "Deployment"
description: "How to build and deploy allocazione.dev as a static site to GitHub Pages or Cloudflare Pages"
---

allocazione.dev is a fully static site generated by SvelteKit with `@sveltejs/adapter-static`. The build output is a plain `build/` directory containing HTML, JS, CSS, and assets — no Node.js runtime is required at serve time.

<Note>
  `@sveltejs/adapter-static` writes all output to the `build/` directory (configured via `pages: 'build'` and `assets: 'build'` in `svelte.config.js`). Any static host that can serve a directory works.
</Note>

## Build configuration

### `svelte.config.js`

```js svelte.config.js
import adapter from '@sveltejs/adapter-static';

const config = {
	kit: {
		adapter: adapter({
			pages: 'build',
			assets: 'build',
			fallback: '404.html',
			precompress: false,
			strict: true
		}),
		prerender: {
			handleHttpError: ({ path, referrer, message }) => {
				// Suppress 404 error for missing favicon
				if (path === '/favicon.png') {
					return;
				}
				throw new Error(message);
			}
		}
	}
};

export default config;
```

Key options:

| Option | Value | Effect |
|---|---|---|
| `pages` | `'build'` | HTML pages are written here |
| `assets` | `'build'` | Static assets are co-located with pages |
| `fallback` | `'404.html'` | Served for unmatched paths by hosts that support it |
| `precompress` | `false` | No gzip/brotli pre-compression — the host handles this |
| `strict` | `true` | Fails the build if any dynamic routes are not fully pre-renderable |

### `vite.config.js`

```js vite.config.js
import tailwindcss from '@tailwindcss/vite';
import { sveltekit } from '@sveltejs/kit/vite';
import { defineConfig } from 'vite';

export default defineConfig({
	plugins: [tailwindcss(), sveltekit()],
	build: {
		rollupOptions: {
			output: {
				manualChunks(id) {
					// Keep all svelte internals in one chunk to avoid circular dep warnings
					if (id.includes('node_modules/svelte/')) {
						return 'svelte-internals';
					}
				}
			}
		}
	}
});
```

All Svelte internal modules are bundled into a single `svelte-internals` chunk to prevent Rollup circular-dependency warnings during the build.

### `.npmrc`

```ini .npmrc
engine-strict=true
```

`engine-strict=true` causes `npm install` to fail if the local Node.js version does not satisfy the `engines` field in `package.json`. This prevents silent version mismatches in CI.

## Build and preview

<Steps>
  <Step title="Install dependencies">
    ```bash
    npm install
    ```
  </Step>

  <Step title="Build the static site">
    ```bash
    npm run build
    ```

    Runs `vite build`, which invokes SvelteKit's prerenderer and writes the fully static output to `./build/`.
  </Step>

  <Step title="Preview the build locally">
    ```bash
    npm run preview
    ```

    Starts a local server serving the `build/` directory. Useful for verifying the production output before deploying.
  </Step>
</Steps>

## Deployment targets

### FTP / managed hosting (current workflow)

The repository includes a GitHub Actions workflow at `.github/workflows/deploy.yml` that deploys to a managed host via FTP on every push to the `prod` branch.

```yaml .github/workflows/deploy.yml
name: Deploy to Hosting via FTP

permissions:
  contents: read

on:
  push:
    branches:
      - prod

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v6
        with:
          fetch-depth: 0

      - name: Setup Node.js
        uses: actions/setup-node@v6
        with:
          node-version: "24"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Build project
        run: npm run build

      - name: Deploy via FTP
        uses: SamKirkland/FTP-Deploy-Action@v4.3.6
        with:
          server: ${{ secrets.FTP_SERVER }}
          username: ${{ secrets.FTP_USERNAME }}
          password: ${{ secrets.FTP_PASSWORD }}
          protocol: ftp
          local-dir: ./build/
          server-dir: /www/
```

Three repository secrets must be set for this workflow to succeed:

| Secret | Description |
|---|---|
| `FTP_SERVER` | Hostname of the FTP server |
| `FTP_USERNAME` | FTP account username |
| `FTP_PASSWORD` | FTP account password |

To trigger a deployment, merge or push to the `prod` branch.

### Cloudflare Pages

<Steps>
  <Step title="Connect the repository">
    In the Cloudflare dashboard, create a new Pages project and connect the GitHub repository.
  </Step>

  <Step title="Set build settings">
    | Setting | Value |
    |---|---|
    | Build command | `npm run build` |
    | Build output directory | `build` |
    | Node.js version | `24` (match the CI workflow) |
  </Step>

  <Step title="Deploy">
    Cloudflare Pages will build and deploy automatically on every push to the production branch.
  </Step>
</Steps>

<Tip>
  Cloudflare Pages natively supports the `404.html` fallback configured in `svelte.config.js`, so SvelteKit's client-side router will handle all navigation correctly.
</Tip>

### GitHub Pages

<Steps>
  <Step title="Add a GitHub Pages workflow">
    Create `.github/workflows/pages.yml` with a job that runs `npm run build` and uploads `./build/` using the `actions/upload-pages-artifact` and `actions/deploy-pages` actions.
  </Step>

  <Step title="Configure the repository">
    In the repository settings, set the Pages source to **GitHub Actions**.
  </Step>
</Steps>

<Warning>
  GitHub Pages does not natively serve `404.html` for SPAs unless you add a redirect workaround. The `fallback: '404.html'` setting in `svelte.config.js` works correctly on Cloudflare Pages and most managed hosts, but may require additional configuration on GitHub Pages.
</Warning>

## Environment variables

No runtime environment variables are required. All site-specific configuration lives in `src/lib/config.js`. The only secrets that exist are the FTP credentials used by the GitHub Actions deployment workflow.

## Security headers

`src/hooks.server.js` sets the following HTTP response headers on every request during prerendering. When deploying to a static host, these headers are injected at build time via SvelteKit's prerenderer — confirm your host serves them correctly or add equivalent rules in your host's configuration.

| Header | Value |
|---|---|
| `X-Frame-Options` | `SAMEORIGIN` |
| `X-Content-Type-Options` | `nosniff` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` |
| `Content-Security-Policy` | See below |

**Content-Security-Policy breakdown:**

```
default-src 'self';
script-src 'self' 'unsafe-inline';
style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
font-src 'self' https://fonts.gstatic.com;
img-src 'self' data: https://github.com https://*.githubusercontent.com
         https://cdn.stats.fm https://*.stats.fm https://i.scdn.co
         https://mosaic.scdn.com https://is1-ssl.mzstatic.com
         https://www.transparenttextures.com;
connect-src 'self' https://beta-api.stats.fm https://api.github.com;
frame-ancestors 'none'
```

The `connect-src` directive explicitly allows outbound fetch requests to the Stats.fm API (`beta-api.stats.fm`) and the GitHub REST API (`api.github.com`), which are the only two external data sources used at runtime.
